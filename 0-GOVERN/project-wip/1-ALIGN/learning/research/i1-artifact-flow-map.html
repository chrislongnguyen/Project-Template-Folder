Agent Pattern Catalog

Sources: Anthropic + Google ADK. Selected during DSBV Design stage.

1. Sequential Pipeline

A → B → C. Linear chain with validation gates.
When: Tasks have clear dependencies. A's output = B's input.

2. Routing / Dispatcher

Classify input → route to specialist sub-agent.
When: Multiple specialist domains. Intent-driven delegation.

3. Parallel Fan-Out / Gather

N agents parallel → synthesizer aggregates.
When: Independent tasks or need diverse perspectives. Speed.

4. Orchestrator-Workers

Central LLM dynamically decomposes → delegates → synthesizes.
When: Subtasks NOT predictable. Dynamic decomposition at runtime.

5. Generator / Critic

Generate → evaluate against criteria → loop until pass.
When: Quality-critical. Binary validation criteria exist.

6. Iterative Refinement

Generate → Critique → Refine. Multi-pass polishing.
When: Qualitative improvement. Polish, tone, optimization.

7. Autonomous Agent

LLM plans independently, tools in loops, human checkpoints.
When: Open-ended exploration. Unknown step count.

8. Human-in-the-Loop

Agent groundwork → pause for human at high-stakes moments.
When: Irreversible actions. Built into DSBV gates (G1-G4).

9. Composite

Mix any of the above. Most production systems.
When: Real-world complexity. Multiple LT risks.

Selection Decision Tree

Q1: Single agent, one context?
 YES → Single agent (60%+ of tasks)
 NO →
Q2: Subtasks known?
 Independent → Fan-Out
 Sequential → Pipeline
 Dynamic →
Q3: Dynamic decomposition?
 YES → Orchestrator-Workers
Q4: Validation loops?
 Binary → Generator/Critic
 Polish → Iterative Refinement
Q5: Open-ended?
 YES → Autonomous Agent
Q6: Multiple needs?
 YES → Composite

ALWAYS: Human gates at G1-G4
ALWAYS: EP-01 (derisk), EP-09 (decompose)

LT → Pattern Quick Ref

LT-1 Hallucination → Generator/Critic
LT-2 Context overflow → Fan-Out (partitioned)
LT-3 Reasoning chain → Pipeline / Hierarchical
LT-5 Plausibility bias → Voting / Refinement
LT-7 Cost → Single agent (simplest)
LT-8 Alignment drift → Human-in-the-Loop + hooks

DSBV onlyDeliver with Version Control ▶

DSBV in EXECUTE — produces working software, not documents

DESIGN:Technical design per feature (data models, APIs, workflows) + Execution Strategy (agent pattern, git plan). Reviewed before coding.
SEQUENCE:Order build: safety first (validation, error handling), then functionality, then optimization. Map tasks → branches/worktrees.
BUILD:Create branches/worktrees per plan. Agents execute per chosen pattern. Each task: implement → self-verify → commit.
VALIDATE:All tests pass. VANA ACs met. PR → human review → merge to iteration branch → worktree cleanup.

DSBV in IMPROVE — produces validated feedback package
