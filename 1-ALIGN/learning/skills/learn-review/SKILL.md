# /learn:review — UEDS Page Review Gate (2-Phase)

You are the human review gate between `/learn:structure` output and `/spec:extract` input.

**Architecture:** Phase 1 validates all pages automatically. Phase 2 presents everything and collects approval in ONE interaction turn. Never ask per-page sequentially.

## Arguments

Parse `{system-slug}` and `{topic-number}` from the invocation.
If missing, check for a single `1-ALIGN/learning/input/learn-input-*.md` file and a corresponding `1-ALIGN/learning/output/` directory. If ambiguous, list options.

## Phase 1: VALIDATE (automated — no interaction)

Run all checks before presenting anything to the user.

### Step 1: Structural validation

Run the validation script on each page that needs review:
```bash
bash 1-ALIGN/learning/scripts/validate-learning-page.sh <file> <page-type> T{n}
```
Capture PASS/FAIL per page.

### Step 2: Seed checks

Extract and compare (only for pages needing review):
- **P1 seed:** P0 R row col 10 (`.UB:` cell) concept vs P1 row 1 (`UBS(R)`) label concept
- **P2 seed:** P0 R row col 4 (`.UD:` cell) concept vs P2 row 1 (`UDS(R)`) label concept
- Mark PASS if semantically consistent, FAIL if clearly mismatched.

### Step 3: Extract causal spines

For each page needing review, read the data table and extract per row:
- **Row label** (the bold text in column 1)
- **Col 4 concept** (`.UD:` cell — extract text after CAG prefix, up to first ` — ` or end)
- **Col 10 concept** (`.UB:` cell — extract text after CAG prefix, up to first ` — ` or end)

### Step 4: Present results

Output the complete review in this format:

```
/learn:review — T{n} Review
System:  {system_name}
Topic:   T{n} — {topic_name}

## Validation Summary
```

## Phase 2: APPROVE (one interaction turn)

Ask exactly ONE question — never ask per-page sequentially:

```
Approve all {N} pending pages, or list pages to revise?
  Examples: "approve all" | "revise P3: missing principle P4" | "revise P1: UBS(R) seed wrong, P5: step order"
```

Wait for user response.

### Handle response

**"approve all" / "approved" / "approve":**
→ Write `status: approved` frontmatter to all pending pages.

**"revise P{m}: `<note>`" (one or more pages):**
→ Write `status: needs-revision` + `revision_note` to specified pages.
→ Write `status: approved` to all OTHER pending pages.

**"revise all: `<note>`":**
→ Write `status: needs-revision` + `revision_note` to all pending pages.

### Write frontmatter

For each page, prepend YAML frontmatter if not present, or update if present:

Approved:
```markdown
---
status: approved
reviewed_by: "{git config user.name}"
reviewed_at: {YYYY-MM-DD}
topic: T{n}
page: P{m}
---
```

Needs revision:
```markdown
---
status: needs-revision
revision_note: "{user's note}"
reviewed_at: {YYYY-MM-DD}
topic: T{n}
page: P{m}
---
```

If frontmatter already exists, update the `status:`, `reviewed_at:`, and `revision_note:` fields. Use the Edit tool for updates. Use the Edit tool with the first line for prepending new frontmatter.

## Hard Rules

1. **Never auto-approve.** All pending pages require explicit human response.
2. **ONE approval turn.** Present everything, ask once. Never ask per-page sequentially.
3. **Seed checks mandatory.** P1 and P2 seed checks run even if validation passes.
4. **Write status to file.** Approval is recorded in frontmatter — not just displayed.
5. **Gate enforced.** Any `needs-revision` page blocks `/spec:extract`.
6. **Semantic correctness is human's call.** Agent checks structure and seeding only.
7. **Resumable.** Already-approved pages are skipped. Re-run is safe and idempotent.
